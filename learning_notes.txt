DAY 1: PROJECT ENVIRONMENT SETUP & PYTHON FUNDAMENTALS
----------------------------------------------------------------------

1. What is a Virtual Environment (venv)?
-----------------------------------------
* In Python, libraries are installed globally on your system by default.
* The Problem: Different projects might require different versions of the
  same library (e.g., FastAPI v0.100 vs. FastAPI v0.111), leading to version
  clashes.
* The Solution: A virtual environment (venv) is an isolated folder inside
  your project directory. It contains its own self-contained copy of Python
  and package installers. Packages installed here do not affect your global
  computer or other projects.
* Commands to create:
  $ python3 -m venv venv

2. Dependency Management (requirements.txt)
--------------------------------------------
* requirements.txt is a standard text file listing every external package
  your project needs.
* It guarantees reproducibility: anyone downloading your code can run a single
  command to install the exact same versions:
  $ venv/bin/pip install -r requirements.txt

3. Text Classification vs. Numeric Regression
----------------------------------------------
To test our general-purpose AutoML agent, we created two different types
of machine learning datasets:
* Spam Dataset (Text Classification):
  - Predicts categories/labels (e.g., "spam" vs. "ham").
  - Requires text preprocessing (removing punctuation, lowercasing) and
    TF-IDF Vectorization (converting words into numbers).
* Weather Dataset (Numeric Regression):
  - Predicts continuous numeric values (e.g., temperature).
  - Requires numeric data scaling/normalization.

4. Python 3.13 & Compatibility Wheels
--------------------------------------
* Python 3.13 is the newest release. Older, strictly pinned libraries
  (like scikit-learn==1.5.0) do not have precompiled binary files ("wheels")
  for Python 3.13.
* When wheels are missing, pip tries to compile packages from C source code.
  If your machine is missing compiler tools (like setuptools providing pkg_resources
  and wheel), the installation crashes.
* Solution: Upgrade pip, setuptools, and wheel, and relax version pins to
  let pip fetch the latest compatible compiled binaries.

DAY 2: BACKEND INFRASTRUCTURE & SQL DATABASES
----------------------------------------------------------------------

1. What is SQLite?
------------------
* SQLite is a lightweight, serverless relational database engine.
* Instead of running a heavy cloud database (like Supabase) or local server,
  SQLite stores all tables in a single local file (e.g., "optima_agent.db")
  inside your project folder.
* It is completely free, secure, fast, and works offline.

2. Database Core Mechanics (Connections & Cursors)
--------------------------------------------------
* sqlite3.connect("optima_agent.db"):
  Establishes a bridge between Python and the database file. If the file
  does not exist, Python automatically creates it.
* row_factory = sqlite3.Row:
  Changes how query results are formatted. Instead of returning values in
  indexes like row[0], Row allows us to access data like a dictionary:
  project["name"]. This prevents indexing bugs.
* Connection vs. Cursor:
  - Connection: The pipeline/bridge to the database.
  - Cursor: The "pen" that writes, runs, and executes SQL commands.
* conn.commit():
  Saves/writes the queries executed by the cursor permanently to the disk.
* conn.close():
  Closes the pipeline to free up the computer's memory.

3. Database Schema Tables
-------------------------
* projects: Stores user projects, CSV paths, target columns, and task types.
* agent_runs: Stores history of every optimization run (model trained, metrics, code).
* settings: Stores private configurations like Gemini API keys.

4. FastAPI Routing & Server Architecture
----------------------------------------
* FastAPI is a high-performance Python web framework for building APIs.
* Routing maps a URL endpoint (like "/api/projects/upload") to a Python function.
* HTTP Methods:
  - GET: Used to retrieve data from the backend database (e.g., listing projects).
  - POST: Used to send data or files to the backend to create records or run tasks.
* BackgroundTasks:
  - ML training is computationally heavy and causes timeouts if run synchronously.
  - BackgroundTasks allows FastAPI to send an immediate "started" response to the
    browser, while continuing to train models silently in a background thread.

5. MLflow Tracking & Script Execution
-------------------------------------
* MLflow UI is a local dashboard that lists all historical runs, metrics, and models.
* It saves all experiments locally in a folder named "mlruns/".
* To launch the UI, we run: mlflow ui --port 5000.
* On macOS, port 5000 is used by AirPlay receiver. Our helper script check if port 5000
  is free using sockets before launching the process.

DAY 3: EXPERIMENT TRACKING & PROCESS MANAGEMENT
----------------------------------------------------------------------

1. What are Network Ports & Port Clashes?
-----------------------------------------
* A port is a numerical "gate" that a computer program uses to listen to web
  traffic (e.g., port 8000 for FastAPI, port 5000 for MLflow).
* Only one program can listen to a port at a time. If another program attempts
  to open on a busy port, it crashes with "Address already in use".
* Apple AirPlay Receiver uses port 5000 on macOS by default, which clashes
  with MLflow's default port.

2. Port Checking with Socket Library
-------------------------------------
* Python's socket library can check if a port is busy:
  socket.socket(socket.AF_INET, socket.SOCK_STREAM) -> Creates a TCP socket.
  connect_ex(('127.0.0.1', 5000)) -> Tries to connect to the port.
  - Returns 0: Connection successful, meaning the port is occupied.
  - Returns non-zero: Connection failed, meaning the port is free.

3. Subprocesses: Blocking vs. Non-blocking
-------------------------------------------
* subprocess.run():
  - Blocking execution.
  - Pauses the parent Python script until the child terminal command finishes.
* subprocess.Popen():
  - Non-blocking execution.
  - Spawns the terminal process in the background and immediately continues
    running the parent Python script.
  - We use Popen for servers (like MLflow) so we can start them and monitor
    them without freezing our script.

DAY 4: THE CODE EXECUTION SANDBOX
----------------------------------------------------------------------

1. What is Sandboxed Code Execution?
------------------------------------
* The AI Agent dynamically writes Python code. We cannot run this code directly
  inside our FastAPI server.
* If the code contains syntax errors, runtime crashes, or infinite loops, running
  it inside the main server would crash the entire API.
* Sandboxing isolates the agent's code inside a separate operating system process
  called a subprocess.

2. Subprocess Execution Mechanics
---------------------------------
We write the code to a temporary file (e.g., "runs/run_xxx.py") and execute it:
* sys.executable:
  Specifies the path to the current running Python binary (inside our "venv").
  This ensures the sandbox runs with the same packages (scikit-learn, pandas).
* stdout=subprocess.PIPE:
  Captures standard console prints (e.g., Accuracy metric printouts).
* stderr=subprocess.PIPE:
  Captures exceptions and tracebacks. This traceback is sent back to Gemini
  as feedback for self-debugging.
* timeout:
  Kills the script if execution exceeds a set duration (e.g., 45 seconds),
  guarding against infinite loops.


----------------------------------------------------------------------
DAY 5: SANDBOX UNIT TESTING
----------------------------------------------------------------------

1. What is Sandbox Unit Testing?
--------------------------------
* Unit testing verifies that our sandboxed subprocess executor behaves correctly
  under positive (successful runs) and negative (crashing/looping scripts) scenarios.
* The test script uses Python's built-in assert statement to ensure keys like
  "success", "exit_code", and "stderr" return exact matches.

2. Testing Scenarios:
---------------------
* Scenario A (Clean Code):
  - Expected: success=True, exit_code=0, stdout contains the print statement.
* Scenario B (Crashes):
  - Expected: success=False, exit_code!=0, stderr contains the traceback error
    name (like "ZeroDivisionError").
* Scenario C (Infinite Loops/Timeouts):
  - Expected: success=False, exit_code=-1, stderr contains the custom timeout
    error string, and the script is terminated within the specified duration limit.

DAY 9: SYSTEM PROMPT ENGINEERING & AGENT RECONSTRUCT
----------------------------------------------------------------------

1. What is System Prompt Engineering?
-------------------------------------
* A System Prompt acts as the "Operating System" or base personality for an AI agent.
* It defines the agent's persona (e.g., Senior Data Scientist), details its operational boundaries, and instructs it on how to handle input data and constraints.
* In OptiAgentML, the prompt dynamically injects project-specific variables such as:
  - Dataset Path ({data_path})
  - Target Column ({target_column})
  - Task Type ({task_type})

2. Designing the ReAct Loop System
----------------------------------
* The agent operates under a Reason-then-Action (ReAct) cycle:
  - THOUGHT: The agent reasons about its next move based on dataset details or feedback from previous executions.
  - ACTION: The action taken (e.g., run_code, finish).
  - CODE: The actual Python code to execute in the sandbox.
* The system prompt enforces a structured JSON structure containing: "thought", "action", and "code".

3. Standardizing Sandbox Output & MLflow Logging
-----------------------------------------------
* The generated code must execute silently in our sandbox, save a model file locally (best_model.pkl), and print execution results as JSON.
* Every code trial is tracked automatically in MLflow. This requires:
  - Importing mlflow and mlflow.sklearn.
  - Creating/logging metrics (e.g. Accuracy/F1 for classification, RMSE/R2 for regression).
  - Saving and logging the trained estimator in MLflow for experiment tracking and reproducibility.

DAY 10: THE REACT AGENT LOOP & SQL RUN LOGGING
----------------------------------------------------------------------

1. Maintaining Chat Session History
-----------------------------------
* A core requirement of ReAct loops is history persistence: the agent must remember
  past decisions (successes, compiler warnings, syntax errors) to self-optimize.
* We utilize Gemini's start_chat() SDK object which maintains the alternation of
  messages (user input -> model response) dynamically, ensuring a clean history.

2. Executing dynamically-generated Code
---------------------------------------
* At each iteration, the agent decides on an action:
  - If action == 'run_code': the orchestrator writes the python code to disk,
    runs it inside the isolated subprocess sandbox, and captures output logs.
  - If action == 'finish': the loop exits successfully.

3. Extracting and Parsing Sandbox Outputs
-----------------------------------------
* The sandboxed script is instructed to print a final JSON metadata line to stdout.
* The loop scans the execution output line-by-line looking for valid curly-braced
  JSON dictionaries.
* Upon identification, it extracts:
  - Validation metrics (accuracy, f1-score, rmse, r2)
  - Selected model estimator type (e.g. Random Forest, SVM)
  - Model hyperparameters
  - MLflow unique run ID

4. Storing Multi-Trial History in SQLite
---------------------------------------
* To feed the visual Next.js frontend, each loop iteration logs database entries
  in our sqlite3 table 'agent_runs'.
* Saved fields include: project_id, run_number, model_name, hyperparameters, 
  metric values, standard output/error console prints, and the localized model path (.pkl).
* If a code run crashes, the stderr traceback is logged in the db and fed back
  to the agent to prompt self-correction on the next iteration.

DAY 11: ROBUST PARSING & COMPILER ERROR FEEDBACK LOOPS
----------------------------------------------------------------------

1. Extracting JSON with Markdown wrappers
-----------------------------------------
* LLMs occasionally enclose their JSON objects inside markdown backticks (e.g. ```json ... ```).
* We implement clean_and_parse_json() using regular expression replacements to strip these syntax blocks.
* We locate the actual JSON boundaries using start_idx = text.find('{') and end_idx = text.rfind('}'), isolating the inner dictionary structure before passing it to json.loads(). This handles extra conversational prefixes or suffixes.

2. Sandboxed Failure Categorization & Feedback Prompts
------------------------------------------------------
* If code execution fails, our feedback loop standardizes responses based on sandbox execution outcomes:
  - Timeout: Informs the model that it generated an infinite loop or excessive computation, prompting it to reduce search space or training cycles.
  - Runtime/Compile Crash: Extracts the stderr Python stack trace and relays it back to the agent as developer-level feedback.
  - Missing Outputs: Handles the case where the code executed cleanly (exit code 0) but failed to output the expected metrics JSON block to stdout. It prompts the model with instructions on formatting print statements.

DAY 12: THE API KEY ROTATION POOL & QUOTA SAFETY
----------------------------------------------------------------------

1. Quota Constraints & Rate-Limiting (HTTP 429)
-----------------------------------------------
* Free API access tokens are subject to strict limits, such as a cap on Requests Per Minute (RPM) or Requests Per Day (RPD).
* In iterative ReAct agent cycles, rapid back-to-back LLM calls trigger rate-limit exceptions (ResourceExhausted status 429), terminating runs prematurely.

2. Architecture of the ApiKeyRotator
------------------------------------
* The rotator class connects to our SQLite database to pull all saved API key strings.
* It exposes a wrapper function execute_with_rotation(func, *args, **kwargs).
* This runs the passed generative SDK method. If it encounters a ResourceExhausted exception:
  1. Cycles the current active key index (current_index = (current_index + 1) % pool_size).
  2. Updates global configurations using genai.configure(api_key=new_key).
  3. Retries the execution up to a maximum index cycle count.
  4. Implements a cooldown sleep duration backoff (e.g. 15 seconds) if all keys are temporarily exhausted.

DAY 13: FASTAPI BACKGROUNDTASKS INTEGRATION
----------------------------------------------------------------------

1. Asynchronous Execution with BackgroundTasks
----------------------------------------------
* Running iterative, compute-heavy machine learning optimization loops inside synchronous HTTP requests blocks the server's main thread, causing 504 Gateway Timeouts.
* FastAPI's BackgroundTasks allows a route to return an immediate response to the client (e.g. status code 202/200 confirming the optimization was triggered) and then execute the long-running agent loop function on a separate worker thread.

2. SQLite Multi-Threaded Database Safety
---------------------------------------
* By default, SQLite database files raise operational errors ("database is locked") when accessed concurrently from multiple threads.
* We configure sqlite3.connect() with timeout=30.0. This instructs the SQLite driver to wait for up to 30 seconds for the database file to unlock before throwing an exception, allowing background threads to write metrics safely alongside frontend read queries.

        <html>
          <body>
            <Navbar />         ← appears on ALL pages
            {children}         ← each page renders here
            <Footer />         ← appears on ALL pages
          </body>
        </html>
      )
    }

3. Client Components vs Server Components
------------------------------------------
* By default in App Router, all components are SERVER components.
  - They run on the server, cannot use useState, useEffect, or browser APIs.
  - They are faster (no JS sent to client for them).

* To make a component a CLIENT component:
  - Add 'use client' as the very first line of the file.
  - This enables React hooks (useState, useEffect, useRef etc.)
  - This enables browser APIs (window, document, localStorage etc.)

  Rule of thumb:
    - Data display with no interactivity → Server Component (no 'use client')
    - Buttons, forms, polls, animations → Client Component ('use client')

4. Next.js <Link> Component
-----------------------------
* Always use <Link href="/dashboard"> instead of <a href="/dashboard">
  for internal navigation.

* Why? <Link> does CLIENT-SIDE navigation:
  - Only fetches the new page's JavaScript bundle
  - Does NOT reload the full browser page
  - Is significantly faster — feels like a SPA (Single Page App)
  - Also prefetches linked pages in the background automatically

  Example:
    import Link from 'next/link';
    <Link href="/dashboard">Go to Dashboard</Link>

5. API Key Security Pattern
-----------------------------
* API keys (like Gemini keys) should NEVER be stored in:
  - localStorage (exposed to JavaScript, XSS vulnerable)
  - Frontend state / React useState (lost on reload)
  - URL parameters (visible in browser history)

* Correct pattern used in OptiAgentML:
  1. User pastes key into a frontend form
  2. Frontend sends key via POST to the backend API
  3. Backend stores key in SQLite database (server-side only)
  4. Backend responds with {configured: true, count: N} — NEVER the key itself
  5. Future requests read the key from the database on the server

6. Polling vs WebSockets
--------------------------
* OptiAgentML frontend uses POLLING to get live experiment updates:
  - setInterval(() => fetchRuns(), 3000) — checks every 3 seconds
  - Simple to implement, works with any HTTP backend
  - Downside: slight delay (up to 3s lag), extra HTTP requests

* Alternative (future upgrade): WebSockets
  - Backend pushes updates to frontend instantly
  - No polling loop needed
  - More complex to set up (requires ws:// protocol upgrade)
  - Better for real-time apps at scale

DAY 20: RUN DETAILS INSPECTOR & DYNAMIC FEATURE IMPORTANCES
----------------------------------------------------------------------

1. Transformed vs Original Feature Mapping
--------------------------------------------
* Preprocessors (One-Hot Encoders, Scalers, Imputers) expand columns
  (e.g., categorical__Sex_female, numeric__Age).
* To retrieve feature importances, the ColumnTransformer exposes
  preprocessor.get_feature_names_out() which maps index positions to
  transformed names.
* By splitting names by the transformer delimiter ('__') and cross-referencing
  with the original CSV columns array, we aggregate expanded indicator weights
  back to original parent feature columns.

2. Model Weight Extraction Rationale
--------------------------------------
* Random Forests/Gradient Boosting models expose tree-node splits via
  estimator.feature_importances_.
* Linear estimators (Logistic Regression, Ridge, SVM) expose parameter
  slopes via estimator.coef_.
* For multi-class linear classifiers (where coef_ is a 2D matrix of shape
  [n_classes, n_features]), we take the absolute mean of weights across
  classes to determine absolute column impact: np.mean(np.abs(coef_), axis=0).

3. Normalizing Feature Influence
..................................
* Coefficients and tree splits have different scales.
* To make them visually consistent, we sum absolute weights and divide
  each by the sum (v / total_w) to normalize the influence scores.
* This renders cleanly as 0-100% influence bars inside the frontend layout.

-------------------------------------

    - Deletes the corresponding database run record from SQLite agent_runs.
* Using double-layered try/except structures ensures that database records
  are still successfully cleaned even if the physical file was already missing
  from disk.

DAY 24: MODEL DRIFT & QUALITY VALIDATION ALARMS
----------------------------------------------------------------------

1. Quality Baselines and Threshold Rules
-----------------------------------------
* Set connection context rules for early anomaly detection:
    - Classification: Alarm triggers if Accuracy < 65% or F1-Score < 65%.
    - Regression: Alarm triggers if R2 Score < 10%.
* These checks execute dynamically during AutoML evaluation on the validation split.

2. Schema-Free SQLite JSON Storage Strategy
---------------------------------------------
* Adding columns to production databases is risky and requires migration scripts.
* Solution: We store validation alarms inside the pre-existing JSON `logs`
  text column alongside metric outputs.
* Under-performing models are flagged with a `"validation_alarm": "Alarm text"`
  key inside this dictionary.
* Logging subqueries:
    - SELECT p.*, (SELECT COUNT(*) FROM agent_runs WHERE project_id = p.id AND logs LIKE '%validation_alarm%') as alarm_count
  This aggregates alarms dynamically and returns a `has_alarm = True` boolean flag
  without modifying table definitions.
* Warning events are simultaneously logged as dedicated rows in the `agent_events`
  history logs using the type `"validation_alarm"`.

3. Frontend Warning Indicators
--------------------------------
* Dashboard: Cards show a glowing warning badge `⚠️ Alarm` next to the status dot.
* Experiments table: The score cell maps `run.logs` and displays a red `⚠️` warning
  sign if validation baselines fail.

SUMMARY OF THE PROJECT :

Imagine you want to bake the perfect cake, but you don't know the exact recipe, oven temperature, or ingredient ratios. Instead of baking 100 cakes one-by-one by hand (which takes days), you use a smart robot kitchen helper. You just dump in the ingredients, and the robot immediately bakes 5 different cakes using different recipes at the same time, ranks them by taste, tells you which one won, and packages the exact recipe for you.

OptiAgentML is that smart robot helper—but for data science and AI.

Here is what it does in simple terms:

1. The Core Problem It Solves
Normally, if a programmer wants to teach a computer to predict something (like predicting House Prices based on square footage, or predicting if a Customer will leave a company based on their usage), they have to write hundreds of lines of code, test many different mathematical models, and guess the best parameters.

OptiAgentML automates this entire process.

2. How a User Uses It (Step-by-Step)
Step 1: Upload a Spreadsheet (CSV) You upload any standard spreadsheet. For example, a list of credit card transactions where some are marked as "Fraud" and some are "Safe".
Step 2: Choose Your Target You tell the system: "I want to predict the column named Fraud."
Step 3: Run AutoML You click "Run AutoML". Behind the scenes, the system automatically trains multiple machine learning models (like Logistic Regression, Decision Trees, and Random Forests) to see which one is the most accurate at predicting fraud.
Step 4: Check the Leaderboard You get a ranked leaderboard showing which mathematical model won (e.g. "Random Forest got 98% accuracy!").
Step 5: Test in the Playground You click a button and a panel slides out. You can type in dummy data (e.g. transaction amount = $5000, location = unknown) and click "Predict" to see if the model flags it as fraud in real-time.
Step 6: Export the Model Once you are happy, you can download the model as a .zip package. It contains the model file and a simple Python file. You can take this zip to any other computer and use it immediately.

3. The "Smart" Features We Added
💡 Gemini AI Insights: If your model accuracy is low, Gemini acts as your virtual Senior Data Scientist. It reads your leaderboard results and writes a custom report explaining why the models did poorly and what features you should change in your data to make it better.
⚠️ Validation Alarms: If a model trains but gets less than 65% accuracy, the system triggers a flashing Alarm badge. This warns you that the model is unreliable.
📊 MLflow Connection: Automatically draws diagnostic charts (like Confusion Matrices showing true vs. false predictions) so you don't have to code them.
In short, it is a complete, automated studio that takes raw data and turns it into a working, tested, and optimized AI predictor in a few clicks.
